Four hours to save fifteen minutes.

I was only trying to get out of UAT. I accidentally solved the part of testing everyone actually hates.

I'm lazy in a specific way. I'll spend four hours building something to avoid fifteen minutes of doing it by hand.

The thing I was avoiding was UAT. But the fifteen minutes I actually saved wasn't the clicking — it was what happened every time the UI changed.

The problem was never writing the tests

Every coding agent can write a browser test now. That part became free somewhere in the last two years. What didn't become free is keeping the tests alive.

Ask anyone who owns an end-to-end suite. Selectors rot the moment a designer moves a button. Runs go red for reasons that aren't bugs. And after the tenth false alarm, the team stops believing red at all — which is the moment a real regression walks straight through.

A test has two lives. The first run needs judgment — something has to figure out where the buttons are and what "done" looks like. Every run after that shouldn't need any judgment at all. The industry keeps paying for the first life on every single run: either frontier-model prices per step, or an engineer's afternoon every time a selector breaks.

So I built the ladder.

Record once. Replay free. Heal on change.

The expensive model runs once. Everything after that only pays for what actually changed:

The trace doesn't store brittle selectors or pixel coordinates — it stores what the agent meant ("the Submit button", "the amount field"), re-resolved against the live page at replay time. So new test data, shuffled record IDs, cosmetic drift: the replay doesn't care. Only a real change in the flow wakes a model up, and it's the free one.

First run: a few minutes and ~70,000 tokens on my own GPU. Every run after: ~90 seconds and nothing.

Which is when I noticed I'd rebuilt a corporate org chart. I delegate to Claude, Claude delegates to Qwen, Qwen delegates to Python. Best team I've ever run.

The model never grades its own homework

Here's the objection every engineer raises about AI testing, and they're right to: the agent hallucinates green. A model watching itself click through a flow will happily declare victory because a toast appeared — while the actual data landed wrong, or didn't land at all.

So in this ladder, the model's job ends when the clicking ends. PASS or FAIL is a boring, deterministic script asserting against the database. Not "did a success message show" — did the records land, with the right values, in the right state.

This isn't theoretical. Early on, the UI would silently record UPI and cheque payments as CASH if the fields were filled in the wrong order. Every screen-level check passed — nice green toast, correct total. The verdict caught it, because the verdict doesn't check the screen. It checks each receipt's mode and amount in the database. The model drives; the evidence decides.

When it fails, you know which side broke

The most expensive part of a red build isn't the failure — it's the twenty minutes deciding whether the test is stale or the app is broken. The ladder answers that as a side effect of its own structure.

If the replay diverged and the healer finished the flow and the verdict passed: stale trace. It re-recorded itself; there's a note in the log; nobody's afternoon was harmed. If the flow can't complete, or the verdict fails: real regression — and the run hands over the exact step, the page state, and what it expected to find. Next on the list: attaching the failing API request and response right beside the DOM state, so "frontend or backend?" is answered in the failure report itself, not in a debugging session after it.

The part I didn't plan: it documents itself

Because the agent screenshots every step it takes, each run leaves behind a click-through record of the journey. Then the support questions started. Cashiers would ask things like:

How do I cancel an incorrect money receipt?

Before, I'd write the steps out or hop on a call myself. Now I forward the question to the agent. It builds a walkthrough for that exact need — pulls the right screenshots, marks the buttons to press, and sends back something you click through instead of read.

Most user docs are written once and quietly go stale the moment the UI moves. This is the opposite — built on the spot, always matching the app as it stands today, shaped around the person who asked. Never out of date, because it's never written in advance.

I didn't set out to build any of this. I was trying to get out of UAT.
